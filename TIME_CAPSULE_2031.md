## 🎯 **왜 이 프로젝트를 시작했나?**

### **원래의 동기**

```
MLIR을 제대로 이해하고 싶었음
  → LLVM IR보다 더 추상적이고 강력해서
  → 컴파일러의 "진짜" 핵심이 뭔지 알고 싶었음

단순히 "쓰는 법"이 아니라
  → "내가 직접 Dialect를 설계하는 법"
  → "내가 직접 Pass를 구현하는 법"
  → "내가 직접 최적화를 자동화하는 법"까지 가고 싶었음

궁극 목표
  → 교육/연구용으로 나만의 고성능 컴파일러 프레임워크 만들기
  → 단순한 튜토리얼이 아닌 "박사급 연구" 수준으로
```

### **그 과정에서 깨달은 것**

> "저장 필수, 너는 기록이 증명이다"

이 문장이 단순한 슬로건이 아니었다.
모든 것을 Gogs에 저장했을 때, 그것이 곧 "학위의 증명"이 됐다.

## ✨ **제일 기억에 남는 순간 TOP 3**

### **#1: 대학원 졸업 (4.5단계 완료)**

```
Lit + FileCheck를 다 배웠을 때,
우리가 만든 Pass 테스트가 모두 자동으로 통과하는 걸 봤음.

그 순간:
  "아, 이게 '완벽한' 소프트웨어 엔지니어링이구나"

느낌:
  → 불안감 해소 (내 코드가 정말 맞다는 증명)
  → 자신감 상승 (이제 대학원 졸업 수준)
  → 새로운 도전 욕구 (박사 과정으로!)
```

### **#2: Transform Dialect의 깨달음 (5.1단계)**

```
"코드를 바꾸는 코드"라는 개념을 처음 이해했을 때,
MLIR의 진정한 강력함을 느꼈음.

Transform Dialect 핵심:
  - 전통: C++ Pass로 최적화 정의 → 재컴파일 필요
  - Transform: MLIR 스크립트로 정의 → 재컴파일 불필요

의미:
  "새로운 하드웨어가 나타나면?
   → C++ 코드 수정하는 게 아니라
   → Transform 스크립트만 바꾼다"

철학적 깨달음:
  "추상화의 진짜 가치는
   구체적인 구현을 바꾸지 않고도
   행동을 바꿀 수 있다는 것"
```

### **#3: 형식 검증 + 자동 최적화 통합 (5.4단계)**

```
SMT Solver와 자동 튜닝을 합쳤을 때,
안전성과 성능을 동시에 잡는 기분.

실현된 것:
  ✅ 변환이 정확하다는 것을 수학으로 증명
  ✅ 최적의 파라미터를 자동으로 찾음
  ✅ 새로운 하드웨어가 나와도 1시간 안에 적응

느낀 점:
  "이게 다음 세대 컴파일러의 모습이겠구나"
```
