# 📦 [초등 1.1] 이름표(SSA)는 단 하나뿐!

> **MLIR의 가장 밑바닥에서 시작하는 대학원 여정**
>
> "저장 필수. 너는 기록이 증명이다."
>
> 이 강의는 복잡한 이론 없이, **상자와 이름표**로 MLIR의 핵심을 이해합니다.
> 대학원 졸업을 향한 첫 번째 기록입니다.

---

## 🎯 오늘 배울 것

단 하나의 규칙입니다:

> **"결과물에 붙이는 이름표는 절대로 겹치면 안 된다!"**

---

## 1️⃣ 상상해 보세요

### 상황 설정

여러분 앞에 **빈 상자**가 여러 개 있습니다.

```
┌─────────────┐
│  (빈 상자)  │
└─────────────┘
```

### 첫 번째 상자에 사과를 넣습니다

```
┌─────────────┐
│   🍎 사과   │  ← 상자 이름: %사과_상자
└─────────────┘
```

우리는 이 상자에 **%사과_상자** 라고 이름표를 붙였습니다.

### 두 번째 상자에 포도를 넣습니다

```
┌─────────────┐
│   🍇 포도   │  ← 상자 이름: %포도_상자
└─────────────┘
```

이번에는 **%포도_상자** 라고 다른 이름표를 붙였습니다.

### 여기서 질문: 무엇이 중요할까요?

만약 포도를 넣은 상자를 **%사과_상자** 라고 부르면?

```
❌ 잘못된 상황
┌─────────────┐
│   🍇 포도   │  ← 상자 이름: %사과_상자  (틀렸다!)
└─────────────┘
```

나중에 누군가 "%사과_상자를 열어줄래?"라고 하면...
- 사과가 나올까요?
- 포도가 나올까요?
- **헷갈립니다!** 😕

---

## 2️⃣ MLIR의 규칙

MLIR도 똑같은 원칙으로 작동합니다.

### 규칙: 이름표는 한 번만!

```
✅ 올바른 MLIR
%0 = 더하기(1, 2)        → %0이라는 상자에 결과 3을 넣음
%1 = 곱하기(%0, 10)      → %1이라는 새 상자에 결과 30을 넣음
%2 = 빼기(%1, 5)         → %2라는 새 상자에 결과 25를 넣음

❌ 틀린 MLIR
%0 = 더하기(1, 2)        → %0이라는 상자에 결과 3을 넣음
%0 = 곱하기(%0, 10)      → %0이라는 상자에 또 다른 것을 넣음 (불가!)
```

### 왜 이런 규칙이 있을까요?

상자와 이름표를 헷갈리지 않으면, **누가 뭘 가져갔는지 정확히 알 수 있기 때문**입니다.

대학원에서는 컴퓨터가 이것을 이용해서:
- "이 상자는 안 쓰니까 버려!"
- "이 계산은 미리 해 두자!"
- "이 일들은 동시에 해도 돼!"

같은 똑똑한 최적화를 할 수 있게 됩니다.

## 3️⃣ MLIR 이름표는 어떻게 생겼나요?

### 형태 1: 번호로 붙이기

```mlir
%0 = ...
%1 = ...
%2 = ...
```

가장 간단합니다.

### 형태 2: 의미 있는 이름으로 붙이기

```mlir
%result = ...
%sum = ...
%product = ...
%apple = ...
```

더 읽기 좋습니다.

### 규칙

모든 이름 앞에는 **% 기호**가 붙습니다.

```
%이름

예시:
%0, %1, %apple, %result, %my_variable
```

## 4️⃣ [실습] 틀린 그림 찾기

### 문제

아래의 두 가지 **설계도** 중에서 **MLIR의 규칙(초등 1.1)**을 어긴 것은 무엇일까요?

#### 설계도 A

```
작업 1: %0 = 더하기(1, 2)      결과: 3을 %0 상자에 넣음
작업 2: %1 = 곱하기(%0, 10)   결과: 30을 %1 상자에 넣음
```

#### 설계도 B

```
작업 1: %0 = 더하기(1, 2)      결과: 3을 %0 상자에 넣음
작업 2: %0 = 곱하기(%0, 10)   결과: 30을 %0 상자에 또 넣음 ← 주목!
```

### 정답

**[설계도 B]가 틀렸습니다!** ❌

왜냐하면 **%0이라는 이름표를 두 번 사용**했기 때문입니다.

### 설명

- 설계도 A: %0, %1 → **다른 이름표** → ✅ 올바름
- 설계도 B: %0, %0 → **같은 이름표 두 번** → ❌ 틀림

## 5️⃣ 더 복잡한 예제로 연습해 보세요

### 예제 1: 과일 주스 만들기

```
작업 1: 사과 두 개를 갈아서 사과 주스를 만듭니다
        %apple_juice = 사과 갈기(사과, 사과)

작업 2: 포도 한 개를 갈아서 포도 주스를 만듭니다
        %grape_juice = 포도 갈기(포도)

작업 3: 두 주스를 섞습니다
        %mixed_juice = 섞기(%apple_juice, %grape_juice)
```

**분석**:
- %apple_juice: 한 번만 정의 ✅
- %grape_juice: 한 번만 정의 ✅
- %mixed_juice: 한 번만 정의 ✅
- **이것은 올바른 MLIR 설계입니다!**

### 예제 2: 계산하기

```
작업 1: %x = 10 + 5        → %x에 15를 저장
작업 2: %y = %x × 2        → %y에 30을 저장
작업 3: %z = %y - 3        → %z에 27을 저장
```

**분석**:
- 각 이름표(%x, %y, %z)는 정확히 한 번씩만 정의됨 ✅

## 6️⃣ 실제 MLIR 코드로 보기

### 예시: 더하기

```mlir
%0 = arith.addi %a, %b : i32
```

**해석**:
- `%0` = 결과물의 이름표
- `arith.addi` = 정수 더하기 연산
- `%a, %b` = 입력값들
- `i32` = 32비트 정수 타입

이 연산은 **%a와 %b를 더한 후, 결과를 %0이라는 상자에 넣는** 것입니다.

### 여러 개를 연쇄적으로

```mlir
%0 = arith.addi %a, %b : i32
%1 = arith.muli %0, 2 : i32
%2 = arith.subi %1, 3 : i32
```

**해석**:
1. `%0` = %a + %b
2. `%1` = %0 × 2 (= (%a + %b) × 2)
3. `%2` = %1 - 3 (= ((%a + %b) × 2) - 3)

각 이름표가 정확히 한 번씩만 나타납니다.

## 7️⃣ 잘못된 예시 (피해야 할 것들)

### 틀린 코드 1: 같은 이름표 두 번

```mlir
❌ %0 = arith.addi %a, %b : i32
❌ %0 = arith.muli %0, 2 : i32      ← %0을 다시 정의!
```

이건 MLIR 규칙을 어깁니다!

### 올바른 코드

```mlir
✅ %0 = arith.addi %a, %b : i32
✅ %1 = arith.muli %0, 2 : i32      ← 새로운 이름 %1 사용
```

## 9️⃣ 왜 이것이 "대학원 연구"에 중요할까요?

### 이 규칙이 있으면 할 수 있는 것들

대학원에서 배우게 될 **컴파일러 최적화**들:

1. **죽은 코드 제거 (DCE)**
   ```
   %0 = ... (이 값을 아무도 안 쓰면)
   → %0을 만드는 연산을 버려버림!
   ```

2. **계산 미리 하기 (Constant Folding)**
   ```
   %0 = 2 + 3
   → 미리 %0 = 5로 계산!
   ```

3. **동시에 할 수 있는 일 찾기**
   ```
   %0 = ... (10번 걸림)
   %1 = ... (20번 걸림, %0을 안 쓰면)
   → %0과 %1을 동시에 할 수 있음!
   ```

이 모든 것이 **이름표 규칙** 덕분에 가능합니다!
