## Part 1: SSA (Static Single Assignment) - 컴파일러의 기초

### 1.1 SSA의 정의와 의미

#### 정의
**SSA (Static Single Assignment)**는 다음 두 가지 조건을 만족하는 중간 표현의 형태입니다:

1. **Every definition dominates all its uses**
   - 변수의 정의가 그 변수를 사용하는 모든 지점을 "지배(dominate)"해야 합니다.

2. **Every variable is assigned exactly once**
   - 프로그램의 정적(static) 실행 구조에서, 각 변수는 정확히 한 번만 정의됩니다.

#### 수학적 표현

```
주어진 프로그램 P와 변수 v에 대해:
- |def(v)| = 1  (정의가 정확히 1개)
- ∀use(v) ∈ uses(v): dominates(def(v), use(v))
  (모든 사용이 정의에 의해 지배됨)
```

### 1.2 왜 SSA가 필요한가? (대학원 연구자의 관점)

#### 이유 1: 데이터 흐름 분석의 단순화

**Non-SSA 코드**:
```python
x = 5         # 정의 1
x = 10        # 정의 2
print(x)      # x의 어느 정의를 사용하는가?
```

컴파일러는 `print(x)`가 어느 `x`를 사용하는지 추적해야 합니다.
→ **복잡한 제어 흐름 분석 필요**

**SSA 형식**:
```
x1 = 5
x2 = 10
print(x2)    # x2가 무엇인지 명확함
```

각 정의가 고유한 이름을 가지므로 **데이터 흐름이 명확**합니다.

#### 이유 2: 최적화의 효율성

SSA 형식에서는 다음과 같은 최적화가 매우 간단해집니다:

**Dead Code Elimination (DCE)**:
```mlir
// SSA 형식
%0 = arith.addi %a, %b : i32   // 이 값이 사용되지 않음
%1 = arith.muli %c, %d : i32   // 사용됨
return %1 : i32

→ %0은 어느 변수도 참조하지 않으므로 제거 가능
  (SSA 덕분에 명확함)
```

#### 이유 3: 병렬화와 의존성 분석

SSA에서는 각 변수가 정확히 한 번 정의되므로:
- 변수 간의 의존성이 명확합니다.
- 병렬 가능한 연산을 쉽게 찾을 수 있습니다.

### 1.3 SSA의 구체적 예시

#### 예시 1: 조건문이 있는 경우

```python
# 원본 코드 (Non-SSA)
if condition:
    x = 5
else:
    x = 10
print(x)
```

**비교: Non-SSA 컴파일러의 고민**
```
조인점(join point)에서:
  print(x)의 x는 어디서 온 것인가?
  - if 브랜치에서 5?
  - else 브랜치에서 10?
  → 제어 흐름 분석 필요
```

**SSA 형식으로 해결**:
```mlir
// if 브랜치
%0 = arith.constant 5 : i32

// else 브랜치
%1 = arith.constant 10 : i32

// 조인점
%2 = scf.if %condition -> i32 {
  scf.yield %0 : i32
} else {
  scf.yield %1 : i32
}
return %2 : i32
```

**PHI 함수 (Phi Node)**를 사용하여 여러 브랜치의 값을 하나로 통합:
```mlir
// 더 저수준 표현에서
bb0:  // if entry
  %0 = arith.constant 5 : i32
  br bb2(%0 : i32)

bb1:  // else entry
  %1 = arith.constant 10 : i32
  br bb2(%1 : i32)

bb2(%arg : i32):  // join point
  // %arg는 %0 또는 %1 중 하나
  return %arg : i32
```

### 1.4 MLIR에서 SSA의 표현

#### 기본 규칙

MLIR의 모든 값은 `%name` 형식입니다:

```mlir
// ✅ 올바른 SSA
%0 = arith.addi %a, %b : i32      // %0 정의
%1 = arith.muli %0, %c : i32      // %0 사용, %1 정의
return %1 : i32                    // %1 사용

// ❌ 잘못된 SSA (같은 이름으로 두 번 정의)
%x = arith.addi %a, %b : i32
%x = arith.muli %x, %c : i32      // ❌ %x 재정의!
```

#### 특수 값: Block Arguments

블록은 PHI 함수와 같은 역할을 하는 **인자(argument)**를 가질 수 있습니다:

```mlir
bb0:
  br bb1(%a : i32, %b : i32)    // bb1에 두 값 전달

bb1(%x : i32, %y : i32):        // %x, %y는 블록 인자
  %z = arith.addi %x, %y : i32
  return %z : i32
```

## Part 2: MLIR의 계층 구조 (Hierarchical Structure)

### 2.1 계층의 개요

MLIR의 전체 구조는 **네 가지 계층**으로 이루어져 있습니다:

```
┌──────────────────────────────────────┐
│ 1. Module (최상위)                   │
│    전체 프로그램 단위                │
└───────────────┬──────────────────────┘
                │
        ┌───────▼──────────┐
        │ 2. Function      │ (기호 테이블의 심볼)
        │    실행 가능 단위│
        └────────┬─────────┘
                 │
         ┌───────▼────────┐
         │ 3. Region      │ (제어 흐름 영역)
         │    블록 묶음   │
         └────────┬───────┘
                  │
          ┌───────▼───────┐
          │ 4. Block      │ (순서있는 연산)
          │    연산 시퀀스│
          └────────┬──────┘
                   │
          ┌────────▼──────┐
          │ 5. Operation  │ (최소 단위)
          │    실제 계산  │
          └───────────────┘
```

### 2.2 각 계층의 상세 설명

#### 계층 1: Module (모듈)

**정의**: MLIR의 최상위 컨테이너. 일반적으로 하나의 컴파일 단위입니다.

```mlir
module {
  // 모든 함수와 심볼이 여기에 포함됨
}
```

**역할**:
- 컴파일 단위 정의
- 전역 심볼 테이블 관리
- 여러 함수의 집합 포함

**속성**:
- 이름 지정 가능: `module @my_module { ... }`
- 메타데이터 추가 가능: `module attributes { ... } { ... }`

#### 계층 2: Function (함수)

**정의**: 실행 가능한 단위. C의 함수와 유사합니다.

```mlir
func.func @add_numbers(%arg0: f32, %arg1: f32) -> f32 {
  %0 = arith.addf %arg0, %arg1 : f32
  return %0 : f32
}
```

**구성 요소**:
- **심볼 이름**: `@add_numbers`
- **입력 인자**: `%arg0: f32, %arg1: f32`
- **반환 타입**: `-> f32`
- **함수 본체**: 하나 이상의 블록

**특수한 함수**:
- `func.func @main() {}` - 진입점
- 외부 함수: `func.func @external_func() : f32` (구현 없음)

#### 계층 3: Region (영역)

**정의**: 제어 흐름 그래프의 영역. 하나 이상의 블록을 포함합니다.

```mlir
// 예: if-else 구조를 표현하는 scf.if Operation
%result = scf.if %condition -> i32 {
  // Region 1 (then 브랜치)
  %0 = arith.constant 5 : i32
  scf.yield %0 : i32
} else {
  // Region 2 (else 브랜치)
  %1 = arith.constant 10 : i32
  scf.yield %1 : i32
}
```

**역할**:
- 제어 흐름의 분기 표현
- 루프 본체, 조건문의 then/else 등을 격리

#### 계층 4: Block (블록)

**정의**: 제어 흐름 없이 순서대로 실행되는 연산들의 모임.

```mlir
bb0:
  %0 = arith.addi %a, %b : i32
  %1 = arith.muli %0, %c : i32
  br bb1(%1 : i32)

bb1(%arg : i32):
  return %arg : i32
```

**특징**:
- 라벨로 식별: `bb0`, `bb1`, ...
- 블록 인자 지원: `bb1(%arg : i32)`
- 분기(terminator)로 끝남: `br`, `return` 등

#### 계층 5: Operation (연산)

**정의**: MLIR의 가장 작은 단위. 실제 계산 수행.

```mlir
%result = arith.addi %a, %b : i32
```

이는 다음 장에서 자세히 다룹니다.

### 2.3 계층 간의 관계 (포함 관계)

```
Module
  ├─ Function
  │    └─ Region (함수 본체)
  │         └─ Block (진입 블록)
  │              ├─ Operation
  │              ├─ Operation
  │              └─ Terminator (br, return, ...)
  │
  └─ Function
       └─ ...
```

**중요한 관찰**:
- 함수는 정확히 하나의 Entry Region을 가집니다.
- Block은 항상 다른 블록으로의 제어 흐름(br)이나 함수 종료(return)로 끝나야 합니다.

## Part 3: Operation의 해부 (가장 중요!)

### 3.1 Operation의 표준 형식

MLIR의 모든 연산은 다음과 같은 표준 형식을 따릅니다:

```
%result = "dialect.op_name"(%operand0, %operand1, ...)
          {attribute_name = "attribute_value"}
          : (input_type0, input_type1, ...) -> output_type
```

혹은 더 간단한 **고수준 형식**으로:

```
%result = dialect.op_name %operand0, %operand1, ...
          {attribute = value} : (input_type) -> output_type
```

### 3.2 각 부분의 상세 설명

#### 부분 1: 결과값 (%result)

```mlir
%0 = arith.addi %a, %b : i32
 ↑
 결과값 (SSA Value)
```

**특징**:
- SSA 형식: 각 결과는 고유한 이름을 가집니다.
- `%0`, `%1`, ... 또는 명시적 이름: `%sum`, `%product`
- 여러 결과값을 가질 수 있습니다 (다중 반환):

```mlir
%x, %y = some.multi_result_op %a : (i32) -> (i32, f32)
```

#### 부분 2: Dialect와 Operation 이름

```mlir
%0 = arith.addi %a, %b : i32
     ↑     ↑
   Dialect Operation
```

**구조**:
- **Dialect**: 연산의 도메인을 나타냅니다.
  - `arith` - 산술 연산 (addi, addf, muli, divf 등)
  - `memref` - 메모리 참조 (load, store)
  - `affine` - 반복문 및 다차원 배열 (affine.for, affine.load)
  - `scf` - 제어 흐름 (scf.if, scf.for)
  - `linalg` - 선형 대수 (linalg.matmul, linalg.gemm)

- **Operation 이름**: 해당 연산의 이름
  - `arith.addi` - 정수 덧셈
  - `arith.addf` - 부동소수점 덧셈

#### 부분 3: 피연산자 (%operand)

```mlir
%0 = arith.addi %a, %b : i32
              ↑  ↑
            피연산자들
```

**특징**:
- SSA 값으로 참조됩니다.
- 순서가 중요합니다 (지정하지 않은 이상).
- 함수 인자, 이전 연산의 결과값 등이 될 수 있습니다.

```mlir
func.func @example(%arg0: i32, %arg1: i32) -> i32 {
  %0 = arith.addi %arg0, %arg1 : i32      // 함수 인자 사용
  %1 = arith.muli %0, %arg1 : i32         // 이전 결과값 사용
  return %1 : i32
}
```

#### 부분 4: 속성 (Attribute)

```mlir
%0 = arith.constant {value = 42 : i32} : i32
                    ↑
                  속성
```

**정의**:
- **컴파일 타임에 결정**된 값입니다. (런타임에 변경 불가)
- Operation의 의미론적 정보를 제공합니다.

**예시**:

1. **상수값**:
```mlir
%0 = arith.constant {value = 42 : i32} : i32
```

2. **루프 범위**:
```mlir
affine.for %i = 0 to 100 {
  // %i는 0부터 99까지
}
```

3. **메모리 접근 패턴**:
```mlir
memref.load %A[%i] {alignment = 64} : memref<?xf32>
```

#### 부분 5: 타입 정보 (Type Signature)

```mlir
%0 = arith.addi %a, %b : i32
                         ↑
                      타입
```

또는 더 복잡한 경우:

```mlir
%0 = some.op %a, %b : (i32, f32) -> tensor<10x20xf32>
              ↑  ↑    ↑                ↑
          operand타입  입력            출력
```

**MLIR의 강 타입 시스템**:
- 모든 값은 타입을 가져야 합니다.
- 입력 타입과 출력 타입이 명시되어야 합니다.

**기본 타입들**:
```mlir
i1, i8, i16, i32, i64           // 정수
f32, f64, f16, bf16             // 부동소수점
tensor<10x20xf32>               // 텐서
memref<10x20xf32>               // 메모리 참조
vector<4xf32>                   // 벡터
```

### 3.3 Operation의 "제너릭" 형식 vs "인쇄" 형식

#### 제너릭 형식 (Generic Form)

```mlir
%0 = "arith.addi"(%a, %b) : (i32, i32) -> i32
```

**특징**:
- 정확하고 엄밀합니다.
- 모든 정보를 명시합니다.
- 파서가 항상 이 형식을 이해합니다.

#### 인쇄 형식 (Pretty Form)

```mlir
%0 = arith.addi %a, %b : i32
```

**특징**:
- 더 읽기 좋습니다.
- 축약된 형태입니다.
- Dialect가 custom parser를 정의해야 합니다.

**변환 가능**:
```
arith.addi %a, %b : i32
↕ (제너릭 형식으로 변환)
"arith.addi"(%a, %b) : (i32, i32) -> i32
```

## Part 4: 실습 및 분석

### 4.1 실습: 기본 코드 분석

다음 MLIR 코드를 분석해 봅시다:

```mlir
module {
  func.func @my_first_op(%arg0: f32, %arg1: f32) -> f32 {
    %0 = arith.addf %arg0, %arg1 : f32
    return %0 : f32
  }
}
```

**분석 문제들**:

1. **모듈의 역할은 무엇인가?**
   - 이 파일 전체를 감싸는 최상위 컨테이너

2. **함수의 인자 `%arg0`와 `%arg1`은 어디서 정의되었는가?**
   - 함수 시그니처에서 정의됨: `(%arg0: f32, %arg1: f32)`

3. **`%0`은 언제 정의되고 언제 사용되는가?**
   - 정의: `%0 = arith.addf ...`
   - 사용: `return %0 : f32`

4. **`arith.addf`의 입출력 타입은 무엇인가?**
   - 입력: `(f32, f32)` (암시적)
   - 출력: `f32`

5. **이 함수는 SSA를 만족하는가?**
   - 예. 각 값이 정확히 한 번만 정의됨.

### 4.2 실습: 더 복잡한 코드 분석

```mlir
module {
  func.func @matrix_ops(%A: tensor<4x4xf32>,
                        %B: tensor<4x4xf32>) -> tensor<4x4xf32> {
    // 요소별 덧셈 (element-wise addition)
    %0 = arith.addf %A, %B : tensor<4x4xf32>

    // 결과에 2를 곱함
    %1 = arith.constant {value = 2.0 : f32} : f32
    %2 = linalg.scale_tensor %0, %1 : tensor<4x4xf32>, f32 -> tensor<4x4xf32>

    return %2 : tensor<4x4xf32>
  }
}
```

**분석**:

1. **SSA 값의 흐름**:
   - `%A, %B` (입력) → `%0` (덧셈 결과) → `%2` (스케일링 후)

2. **타입 일관성**:
   - `%A: tensor<4x4xf32>`
   - `%B: tensor<4x4xf32>`
   - `%0: tensor<4x4xf32>` (덧셈 결과)
   - `%1: f32` (상수)
   - `%2: tensor<4x4xf32>` (최종 결과)

3. **Dialect의 역할**:
   - `arith.*` - 기본 산술 연산
   - `linalg.*` - 선형 대수 연산

### 4.3 실습: 조건문과 Control Flow

```mlir
func.func @conditional_add(%cond: i1, %a: i32, %b: i32) -> i32 {
  %0 = scf.if %cond -> i32 {
    // then 브랜치
    %1 = arith.addi %a, %b : i32
    scf.yield %1 : i32
  } else {
    // else 브랜치
    %2 = arith.subi %a, %b : i32
    scf.yield %2 : i32
  }

  return %0 : i32
}
```

**분석**:

1. **Region의 역할**:
   - `scf.if`는 두 개의 Region을 가집니다 (then과 else)
   - 각 Region은 하나의 Block을 포함합니다.

2. **PHI 함수의 역할**:
   - `%0`은 `%1` 또는 `%2` 중 하나를 받습니다.
   - 어느 것을 받을지는 `%cond`에 의해 결정됩니다.

3. **Terminator의 역할**:
   - `scf.yield`는 블록의 끝을 나타냅니다.
   - 값을 외부로 전달합니다.

### 4.4 **대학원 연구 문제** (심화)

**문제 1**: 다음 코드는 SSA를 만족하는가?

```mlir
func.func @loop_sum(%n: i32, %values: memref<?xi32>) -> i32 {
  %init = arith.constant {value = 0 : i32} : i32
  %0 = scf.for %i = %init to %n step 1 iter_args(%sum = %init) -> i32 {
    %val = memref.load %values[%i] : memref<?xi32>
    %new_sum = arith.addi %sum, %val : i32
    scf.yield %new_sum : i32
  }
  return %0 : i32
}
```

**분석**:
- `%init`: 초기값 정의
- `scf.for` 루프에서:
  - `%sum`: 루프 반복 변수 (각 반복마다 다른 값)
  - `%new_sum`: 새로운 합 계산
  - `scf.yield %new_sum`: 다음 반복으로 전달
- `%0`: 루프 최종 결과

→ **예. SSA를 만족합니다.** (모든 값이 정확히 한 번만 정의됨)

## Part 5: 핵심 개념 정리

### 5.1 SSA와 컴파일러 최적화

SSA 덕분에 다음이 가능합니다:

| 최적화 | 설명 | SSA 없이 |
|--------|------|---------|
| Dead Code Elimination | 사용되지 않는 연산 제거 | 복잡한 사용 분석 필요 |
| Constant Folding | 상수 계산 미리 수행 | 변수 재할당 추적 필요 |
| Common Subexpression Elimination | 중복 계산 제거 | 변수 별명(alias) 분석 필요 |
| Loop Invariant Hoisting | 루프 불변식 밖으로 이동 | 의존성 추적 어려움 |

### 5.2 MLIR의 구조적 특징 (정리)

| 계층 | 목적 | 예시 |
|------|------|------|
| Module | 컴파일 단위 | 하나의 `.mlir` 파일 |
| Function | 실행 가능 코드 | `func.func @add(...)` |
| Region | 제어 흐름 분기 | `scf.if`의 then/else |
| Block | 순차 연산 | 분기 없는 연산들 |
| Operation | 최소 계산 단위 | `arith.addi`, `memref.load` |

### 5.3 Operation 문법의 3가지 필수 요소

```mlir
%result = dialect.op_name %operands : (input_types) -> output_type
          ↑                ↑           ↑                 ↑
        SSA Value     Dialect+Op   입력 타입          출력 타입
```
