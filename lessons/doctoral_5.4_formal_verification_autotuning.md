## 🔬 Translation Validation: 변환의 정확성

### 2️⃣ SMT Solver를 이용한 검증

#### 개념

```
최적화 전 코드:
  for i in 0..100:
    C[i] = A[i] + B[i]      (간단한 반복)

최적화 후 코드:
  for i in 0..100 step 4:
    C[i] = A[i] + B[i]      (언롤링)
    C[i+1] = A[i+1] + B[i+1]
    C[i+2] = A[i+2] + B[i+2]
    C[i+3] = A[i+3] + B[i+3]

질문: 두 코드가 동일한 결과를 생성하는가?

답변: SMT Solver가 자동으로 증명
```

#### 검증 과정

```
Step 1: 코드를 논리식으로 변환
  원본: for i: C[i] = A[i] + B[i]
  논리식: ∀i ∈ [0, 99]: C_orig[i] = A[i] + B[i]

  최적화: for i step 4: C[i...i+3] = A[i...i+3] + B[i...i+3]
  논리식: ∀i ∈ {0,4,8,...,96}:
            C_opt[i] = A[i] + B[i]
            C_opt[i+1] = A[i+1] + B[i+1]
            C_opt[i+2] = A[i+2] + B[i+2]
            C_opt[i+3] = A[i+3] + B[i+3]

Step 2: 동등성 검증 (SMT Solver)
  쿼리: C_orig[i] ≠ C_opt[i]인 i가 존재하는가?

  Z3 Solver가 자동 계산:
    → "존재하지 않음" (Unsat)
    → "따라서 두 코드는 의미론적으로 동등"
    → ✅ 변환은 안전함 (Sound)

Step 3: 결과
  "형식적으로 증명됨"
  모든 가능한 입력 A, B에 대해 정확함을 보장
```

#### 현실적 예제

```
최적화: MatMul 타일링

원본 (정의):
  C[i,j] = Σ(k=0..999) A[i,k] × B[k,j]

타일화:
  for k_tile = 0 to 1000 step 256:
    for i_local = 0 to 256:
      for j_local = 0 to 256:
        for k_local = 0 to 256:
          C[...] += A_tile[...] × B_tile[...]

검증:
  "모든 i,j,k에 대해
   타일화된 결과 = 원본 결과

   를 SMT Solver가 증명했으므로,
   타일 크기에 관계없이 정확함이 보장된다."

결과:
  ✅ 이 최적화는 Sound (오류 없음)
  ✅ 박사 논문에 인용 가능
```

## 🤖 Cost Model: 자동화된 의사결정

#### 문제 상황

```

  → 하드코딩 (모든 상황에 적용 불가)

박사급:
  "하드웨어와 문제의 특성을 학습한
   AI 모델이 실시간으로 최적 파라미터 결정"
  → 자동화 (어떤 상황에도 적응)
```

#### Cost Model 구축

```
입력 벡터:
  - 루프의 반복 횟수
  - 메모리 접근 패턴 (stride, reuse)
  - 캐시 크기 (L1, L2, L3)
  - 명령어 레벨 병렬성 (ILP)
  - 분기 예측 실패율

AI 모델:
  Neural Network (Deep Learning)
    입력 벡터 (위의 7개 특성)
         ↓
    Hidden Layer 1 (128 뉴런)
         ↓
    Hidden Layer 2 (64 뉴런)
         ↓
    출력 (최적 언롤 팩터: 1~16)

학습 데이터:
  하드웨어에서 직접 실행하여
  1,000개 프로그램 × 100개 파라미터 조합
  = 100,000개의 (특성, 성능) 쌍으로 학습

결과:
  "새로운 루프가 들어오면
   모델이 0.1초 내에 최적 파라미터 제안"
```

#### 성능 향상

```
수동 튜닝:
  프로그래머가 실험 → 10시간 소비

자동 비용 모델:
  AI가 학습한 모델 사용 → 0.1초

효과:
  - 개발 시간: 10시간 → 0.1초 (10,000배)
  - 최적성: 95% 이상 (최적보다 약간 낮음)
  - 일반화: 새로운 하드웨어도 자동 적응
```

## 🔍 Search Space Exploration

### 4️⃣ 수만 가지 조합의 자동 탐색

#### 문제: Explosion of Combinations

```
최적화 결정:
  - Unrolling: 1, 2, 4, 8, 16 (5가지)
  - Tiling: 32, 64, 128, 256 (4가지)
  - Vectorization: on/off (2가지)
  - Fusion: yes/no (2가지)
  - Parallelization: 1, 2, 4, 8, 16 thread (5가지)

총 조합: 5 × 4 × 2 × 2 × 5 = 400가지

더 복잡한 경우:
  10개 최적화 × 10개 파라미터 = 10,000,000,000가지

사람이 전부 테스트? 불가능!
→ 자동 탐색 프레임워크 필요
```

#### 자동 탐색 알고리즘

```
Method 1: 무작위 탐색 (Random Search)
  알고리즘:
    for i = 1 to 1000:
      params = random_selection(all_params)
      performance = evaluate(params)
      if performance > best:
        best = performance

  장점: 간단함, 빠름
  단점: 최적에서 멀 수 있음

Method 2: 베이지안 최적화 (Bayesian Optimization)
  알고리즘:
    1. 몇 가지 시도 (10개)
    2. 성능 패턴 학습
    3. 다음에 시도할 후보 예측
    4. 가장 유망한 것부터 시도
    5. 반복

  장점: 적은 시도로 좋은 해 찾음
  단점: 복잡함

Method 3: 진화 알고리즘 (Genetic Algorithm)
  알고리즘:
    1. 임의 개체 100개 생성
    2. 성능 평가
    3. 상위 20개 선택
    4. 교배 & 돌연변이로 새 개체 생성
    5. 반복

  장점: 글로벌 최적에 가까움
  단점: 많은 평가 필요
```

#### 박사급 구현

```
하이브리드 탐색 전략:
  Phase 1: Random Search (빠른 탐색)
    100개 임의 조합 평가

  Phase 2: Bayesian Optimization (정밀 탐색)
    50개 유망 조합 평가

  Phase 3: Local Search (미세 조정)
    현재 최적에서 인접한 조합만 평가
    10개 평가

총 160번 평가로 1,000,000,000가지 중
90% 이상의 최적성 달성!

시간: 약 1시간 (GPU 병렬화로 10분까지 단축 가능)
```

## ⚖️ AffineConstraint와 메모리 검증

### 5️⃣ 메모리 경계를 넘지 않음을 증명

#### 문제

```
타일링된 루프:
  for i_tile = 0 to 1000 step 256:
    for i_local = 0 to 256:
      access_memory(i_tile + i_local)

질문: 모든 접근이 배열 경계 [0, 999] 내인가?

직관:
  "i_tile = 0..768 (step 256)
   i_local = 0..255
   → 최대값 = 768 + 255 = 1023
   → 1000 초과! (범위 오버)"

이 직관이 정확한가? 증명해야 함!
```

#### 형식적 증명

```
Affine Constraint System:
  i_tile ∈ {0, 256, 512, 768}
  i_local ∈ [0, 255]

  접근 주소: i = i_tile + i_local

  제약:
    i = i_tile + i_local
    0 ≤ i_tile ≤ 768, i_tile % 256 = 0
    0 ≤ i_local ≤ 255

  질문: 0 ≤ i ≤ 999인가?

  증명 (SMT Solver):
    최악: i_tile = 768, i_local = 255
    → i = 1023 > 999

    → 메모리 범위 초과!
    → 버그 발견 ✓

수정:
  조건 변경:
    for i_tile = 0 to 744 step 256:
    (768 → 744로 변경)

    최악: i_tile = 744, i_local = 255
    → i = 999 ≤ 999 ✓

    SMT가 다시 증명:
    → 안전함 ✓
```

### 6️⃣ 심사위원이 평가하는 기준

#### 1. Correctness (정확성)

```
질문: "당신의 최적화가 프로그램의 논리를 깨뜨리지 않습니까?"

답변 방식:

  ② 메모리 안전성 (Memory Safety)
     "AffineConstraint로 모든 메모리 접근이
      배열 경계 내임을 증명했습니다."

  ③ 정수 오버플로우 검증
     "32비트 정수 연산에서 오버플로우 불가능함을
      범위 분석으로 증명했습니다."

증거:
  - 증명 스크립트 (Coq, Isabelle 등)
  - SMT 검증 리포트

#### 2. Generality (일반화)

```
질문: "특정 벤치마크(ResNet, VGG)에만 잘 작동하는 것 아닙니까?"

답변 방식:
  ① 다양한 도메인 테스트
     - 컴퓨터 비전 (CNN)
     - 자연어 처리 (Transformer)
     - 과학 계산 (행렬 곱셈, FFT)
     - 신호 처리 (필터)

  ② 자동화된 비용 모델
     "ML 기반 비용 모델이 새로운 도메인의
      특성을 자동으로 학습하므로,
      미지의 워크로드에도 적응합니다."

  ③ 다양한 하드웨어
     - NVIDIA GPU (여러 세대)
     - AMD GPU
     - CPU (x86, ARM)
     - 커스텀 가속기

증거:
  - 벤치마크 결과 표 (15개 이상)
  - 새로운 하드웨어 추가 시 자동 적응 증명
  - 신규 도메인에서의 성능

결과:
  ✅ "이 기법은 일반적 (General)"
```

#### 3. Efficiency (효율성)

```
질문: "기존의 최신 기술(SOTA)과 비교하면 얼마나 우수합니까?"

답변 방식:
  ① 성능 비교
     | 방법 | MatMul | Conv2D | RNN | 평균 |
     |------|--------|--------|-----|------|
     | SOTA | 100%   | 100%   | 100%| 100% |
     | 우리 | 115%   | 122%   | 108%| 115% |

  ② 컴파일 시간
     SOTA:     1시간 (수동 튜닝)
     우리:     10분  (자동 튜닝)
     → 6배 빠름

  ③ 에너지 효율
     SOTA:     10W 평균
     우리:     7W 평균
     → 30% 에너지 절감

증거:
  - 성능 테이블
  - 런타임 비교
  - 전력 측정 결과
  - 통계적 유의성 검증

결과:
  ✅ "이 기법은 최신 기술을 능가 (SOTA+)"
```

  → AutoTuning

신뢰성 (Reliability):
  "모든 입력에 대해 정확함을 보장"
  → Formal Verification
  → Memory Safety
  → Integer Overflow Prevention

박사 논문 3축:
  → Correctness (정확함을 증명)
  → Generality (다양한 상황에 적용)
  → Efficiency (최신 기술을 능가)
```
